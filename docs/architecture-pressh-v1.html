Executive Summary

Pressh is a self-hosted, no-code content management system built end-to-end in TypeScript, positioned as a security-first alternative to WordPress. Its thesis is blunt: "WordPress, but the data doesn't leak." Every architectural decision is filtered through the failure modes that make WordPress dangerous — plugin RCE, SQL injection in plugin code, IDOR via guessable IDs, REST user enumeration, and file-upload RCE. Pressh neutralizes these by running all plugins in isolated worker threads behind a default-deny capability gate, never executing third-party code in the render tree, and baking security into the core SDK so plugin authors cannot forget it. The public site and admin studio run as two independent processes on a unified Hono + Vite (SSR) stack — no Next.js — using a WordPress-style front-controller so new content types, pages, and plugin endpoints go live at request time with zero rebuild.

★ RECOMMENDED

Option A — Two-Process Trust-Split

Separate Studio (admin) and Site (public) processes, each with its own plugin host and worker pool. Why: it gives the strongest blast-radius isolation — a compromised public plugin can never reach an admin session — which is the product. Both processes can co-locate on one box, so small installs pay almost no ops tax.

Architecture Options

All three keep the locked decisions (Hono+Vite, TS monorepo, worker-isolated plugins, file-default storage). They differ on process topology — which changes isolation, ops simplicity, and performance.

★ RECOMMENDED

Option A — Two-Process Trust-Split

"Admin and public never share a process."

Studio (admin) and Site (public) run as separate OS processes, each booting its own PluginHost and worker pool, both reading the same /content + /plugins on disk but never sharing memory. A reverse proxy routes /admin to Studio (lockable behind VPN/SSO/IP allowlist) and everything else to Site. State lives entirely in storage, so either process can be restarted or horizontally scaled independently. For a small install both processes run on one host under a process manager — the isolation is logical, not necessarily physical.

Pros
✓ Strongest blast radius isolation — public compromise can't touch admin sessions/cookies
✓ Independent scaling: site scales out, studio stays single small node
✓ Studio can be firewalled off the public internet entirely
✓ Crash/restart of one process never degrades the other
✓ Matches the security thesis — no shared trust boundary to subvert
Cons
✗ Two processes to supervise (mitigated: one compose file / process manager)
✗ Slightly higher baseline RAM (two worker pools)
✗ Shared storage needs file-locking discipline for concurrent writes
✗ Cache invalidation must cross the process boundary (pub/sub or storage-watch)

Option B — Single-Process Modular Monolith

"One Hono server, admin and public separated by middleware."

A single Hono process serves both /admin and public routes, with one shared PluginHost and worker pool. Route namespacing + middleware enforce authz. Simplest possible deploy: one container, one port, one thing to watch. Worker isolation for plugins still holds — but admin and public requests share the host process and the same plugin workers.

Pros
✓ Simplest ops — one process, lowest RAM, easiest local dev
✓ No cross-process cache invalidation problem
✓ Fastest path to a running MVP
✓ Single shared worker pool = better plugin-worker utilization
Cons
✗ Weaker trust boundary — a host-process bug exposes admin + public together
✗ Can't firewall admin separately from public traffic
✗ Public-traffic spikes can starve admin responsiveness
✗ Undercuts the core security differentiator

Option C — Static-First Hybrid

"Pre-render published pages to static; hydrate dynamic bits via API."

On publish, the engine renders content to static HTML written to disk/CDN. The public "site" is mostly static files; dynamic features (search, forms, comments, personalization) call the API process. Studio + API run like Option A. Smallest possible public attack surface (static files can't be injected), best raw performance.

Pros
✓ Best performance & cacheability; trivial CDN fronting
✓ Tiny public attack surface — mostly static assets
✓ Public tier can survive even if the app process is down
Cons
✗ Build/publish step adds latency between "publish" and "live"
✗ Truly dynamic / personalized / logged-in pages are awkward
✗ Large sites = large static rebuilds; incremental rebuild is complex
✗ Instant preview & A/B harder; more moving parts than B

  • Identity: argon2, sessions in httpOnly cookies, MFA-ready, RBAC capabilities, lockout
  • Supply chain: plugin signing, CVE feed, dependency pinning, SBOM in CI

Secrets management

A sealed vault (SecretsBackend interface) ships with a file-based AES-256-GCM backend keyed by PRESSH_MASTER_KEY (provided via env/secret-manager at boot, never written to disk in plaintext). Plugins never see process.env; they request a secret by name and receive a short-lived scoped token only if their manifest holds the matching capability. Rotation: re-encrypt on key roll; audit log records every secret access.

Scalability & Performance

Target: SMB & publishers — thousands to low-millions of monthly views per site.

Per-component scaling strategy

ComponentStrategy
Site processStateless → horizontal scale behind proxy; all state in storage
Studio processStays single small node (admin traffic is tiny); vertical if needed
Render cacheTag-based cache + CDN edge; invalidate by content tag on publish
Plugin workersPool per process; cap count; cache plugin output to avoid repeated RPC
Storage (FS default)SQLite index for queries; migrate to Postgres adapter past ~100k entries
MediaLocal disk → S3-compatible object store + CDN as volume grows

Capacity model

  • Launch (1 VM, 2 vCPU / 4 GB): ~200–500 req/s cached, ~30–60 req/s SSR-uncached, comfortably serves a site at low-millions of monthly views with CDN in front.
  • 2-year (3× site nodes + managed Postgres + CDN): ~1.5k+ req/s SSR, multi-site portfolio scale; bottleneck shifts to DB writes, not render.
  • Editorial: dozens of concurrent editors on one studio node is fine — admin is low-QPS, capability checks are in-process.

Bottleneck map (in order)

  1. SSR CPU on cache miss → tag-cache + CDN + stale-while-revalidate
  2. Plugin RPC latency on plugin-heavy pages → batch RPCs, memoize plugin output per content version
  3. FS write contention under concurrent publish → SQLite index + switch to Postgres adapter
  4. Media bandwidth → object store + CDN offload

Caching philosophy

Public pages are cached by content tag, not by time. A page caches indefinitely and is purged the instant its content (or a dependency) changes via revalidateTag — so editors see instant-live updates and visitors get CDN-speed. Personalized/logged-in responses bypass cache. Plugin output is memoized per content revision to keep worker RPC off the hot path.

RTO / RPO & DR

  • RPO: ≤ 24h with daily backups; ≤ 1h with hourly snapshots / DB WAL
  • RTO: single-node restore ~15–30 min from backup volume
  • DR posture: active-passive — nightly off-host backup of /content, /media, vault, DB; restore to a fresh host
  • Stateless site processes mean HA = run ≥2 behind the proxy

Observability stack

  • Logs: structured JSON (pino), sensitive fields redacted
  • Metrics: Prometheus — req rate, SSR p50/p99, cache hit %, worker restarts
  • Tracing: OpenTelemetry across proxy → app → worker RPC
  • Alerting: SLO breach (p99, error rate, worker crash loop) → runbook links
  • Audit log doubles as a security/compliance signal

Runbook sketches (top 3)

  • Worker crash-loop for plugin X: check worker logs → if reproducible, disable plugin via admin (capability revoke) → site degrades gracefully (block renders fallback) → file issue with plugin author.
  • Public site 5xx spike: check health + SSR p99 → confirm cache hit % drop → warm cache / scale site nodes → if storage-bound, check disk + index integrity.
  • Suspected breach / anomalous capability use: consult audit log → rotate PRESSH_MASTER_KEY + sessions → quarantine offending plugin → restore from known-good backup if needed.

Honest limitations

  • What it doesn't handle well: very plugin-heavy pages pay RPC latency; truly massive single sites will outgrow the FS default and need the DB adapter early; it's heavier to operate than WordPress's "drop files on shared hosting."
  • Assumptions that would invalidate the recommendation: if you actually need multi-tenant SaaS, the trust model and data layer change substantially (revisit before Phase 2). If plugin ecosystem velocity matters more than safety, the isolation tax may feel too high.
  • With double the budget: add a managed control plane, hosted SaaS tier, a vetted plugin marketplace with automated security review, and multi-region active-active.
  • With half: ship Option B (single-process monolith) for MVP and split into two processes later — accept the weaker trust boundary temporarily, document it loudly.
